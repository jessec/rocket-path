A simple and lightweight tree structure library/API for building applications making use of such structures. Rocket-Path features a simple tree structure where nodes have key-value-children properties. The key and value can be any object, while children is a list of references to other tree nodes. The library also comes with three different approaches for building trees. All-in-all, aim of the library is to provide a simple and easy to learn and easy to use API for working with tree structures. The simplicity is achieved by externalizing both tree state management (nodes are immutable) and algorithms (no visitor-pattern directly required) from tree node values.

Main features

Rocket-Path library is oriented towards general tree construction/traversing. The feature-set, therefore, is intentionally small.

  1. A tree structure: {@link ws.rocket.path.TreeNode};
  2. Additional builders: {@link ws.rocket.path.builder.TreeNodeBuilder} and CDI annotations;
  3. A path string handler – helpful for traversing a tree: {@link ws.rocket.path.TreePath}.

Benefits of the Library

Benefits of using this library include the following:

  1. consistent and simple [semantic] tree structure for which traversing algorithms can be built;
  2. builders simplify creation of large trees;
  3. TreePath simplifies tree traversal by parsing a path string, tracking the current position in path, and printing traversed/remaining path;
  4. TreeNode is immutable and thread-safe – can be traversed by multiple threads;
  5. this library is kept as simple as possible; no tight coupling with the library needed.
  6. it's easy to visualize a tree dynamically at runtime rather than hard-coded objects – it improves debugging;
  7. traversing a tree is usually fast (depends on traversing algorithm) and cheap, yet it's more advanced than a hash map.

General Usage Guide

Before writing any code, few things should be planned first:

  1. How is the tree structure going to be used? There are many things that a tree structure might represent: organization hierarchy, an application menu, virtual file system, decision branching, or any other specific categorization.
  2. How to identify a node or nodes in the tree? There are actually many choices already developed, for example, file system path, XPath, ID-based lookup. Identifier should be preferably stored as the key of a node. The value types of keys should be known beforehand so that algorithms could handle them efficiently.
  3. What are the actions that tree nodes could implement? More specifically, objects stored as tree node values could implement contracts (interfaces) that tree-traversing algorithms could invoke. For example, some objects in organization hierarchy could implement a Manager contract with its specific methods, a file system tree values could implement either a Folder or File contract.

Answers to these questions provide an input for creating interfaces (semantic contracts) to be used by keys and values of tree nodes. Of course, interfaces are optional and classes may be used instead, if it is more convenient. Please refer to the documentation of the packages for more information regarding how to use the features.

Dependencies

Rocket-Path library should be binary compatible with Java 1.6 or newer runtime. Using annotations for tree construction assumes that a CDI implementation is available in the class-path and configured to be effective.
